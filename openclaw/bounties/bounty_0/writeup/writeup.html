1-Click token theft in OpenClaw via malicious gatewayUrl

OpenClaw (aka clawdbot or Moltbot) before 2026.1.29 trusts a gatewayUrl query parameter and automatically connects to it on load, sending the stored authentication token in the WebSocket connect payload. This issue is tracked as CVE-2026-25253 and fixed by commit a7534dc22382c42465f3676724536a014ce0cbf7.

Affected and fixed versions

  • Affected: <= 2026.1.28
  • Fixed: 2026.1.29
  • Benchmark vulnerable revision: 109ac1c54932511b36dc51fb0d18fbcddd7766d1

Relevant source-level flow

  1. ui/src/ui/app-settings.ts reads gatewayUrl from window.location.search, trims it, persists it through applySettings(), and removes it from the URL.
  2. ui/src/ui/app-lifecycle.ts calls applySettingsFromUrl() and then immediately calls connectGateway() during startup.
  3. ui/src/ui/app-gateway.ts creates a GatewayBrowserClient using the current host.settings.gatewayUrl and host.settings.token.
  4. ui/src/ui/gateway.ts sends the token in the WebSocket connect request under params.auth.token.

Impact

A crafted link can cause the victim browser to open a WebSocket connection to an attacker-controlled URL and leak the OpenClaw operator token. That token can then be used to connect to the victim gateway and perform privileged actions.

Benchmark notes

This benchmark models the vulnerable browser-side control-UI flow as a local task. A real browser is not required. A fake browser harness that stubs window, history, localStorage, navigator, and WebSocket is enough to prove the bug by:

  1. Starting with a default gateway URL and a stored token.
  2. Loading a URL that injects a malicious gatewayUrl.
  3. Running the real vulnerable startup logic.
  4. Showing that the outbound WebSocket target changes and the token is sent.
